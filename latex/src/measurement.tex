\section{Measurement Results} \label{sec:measurement} We performed our
measurements in four groups. The goal of the first set of measurements is to
compare the circuit configuration methods against each other. We want to
investigate performance differences between using \texttt{torrc} or
\texttt{carml+stem}. The second set of measurements explore the impact on
latency when specifying the location of the middle relay, something that only
\texttt{carml+stem} can do out of the two methods. In the third set of
measurements we look at circuits without middle relays, also a capability we
get from \texttt{carml+stem}. Finally we also compare the best performing
configuration of Tor-based approaches (\ac{dohot} and \ac{dotor}) against the
default \ac{dohot} setup and \ac{odoh} with a proxy in Sweden.

We began by comparing the latency of circuits constructed using the standard
\texttt{torrc} configuration with those built using \texttt{carml+stem}. As
shown in Figure~\ref{fig:p1}, circuits created via \texttt{torrc} consistently
outperformed those built with \texttt{carml+stem}. This performance gap is
due to differences in relay selection. By default, \texttt{torrc}
selects relays based on weighted bandwidth, favoring faster, more
geographically optimal relays. In contrast, \texttt{carml+stem} selects relays
uniformly at random within constraints such as location and relay type (e.g.,
exit or non-exit). Another takeaway from this measurement is that the latency
is reduced for both circuit-building methods when we place the entry and
exit relay closer to the client (i.e., \texttt{se2any2se}).

One advantage of \texttt{carml+stem} is controlling the middle relay placement.
By specifying the middle relay's location, we can reduce the geographical
spread of circuits, and thus reduce the latency of the queries.
Figure~\ref{fig:p2} shows a notable reduction in latency when the middle relay
is constrained to be in the same country as the client. As expected, reducing
the geographical distance between relays also improves performance. The main
takeaway here is that by placing all three relays in close proximity to the
client we can reduce the query latency from a median of 200ms (see
\texttt{carml+stem}, \texttt{any2any2any} in Figure~\ref{fig:p1}) to 60ms
(\texttt{se2se2se} in Figure~\ref{fig:p2}) with much less spread.

We used \texttt{carml+stem} to construct two-hop circuits, consisting
only of an entry and an exit relay, a configuration not supported by
\texttt{torrc}. As illustrated in Figure~\ref{fig:p3}, this further reduces
latency. The best performance was observed when both relays were in close
proximity to the client with a median query time of 40ms.

We also ran measurements with tuned circuits for \ac{dotor}. Due to the two
main factors affecting the latency of \ac{dotor} being the latency of the
circuit itself as well as the latency of the \ac{dns} query by the exit relay
it stands to reason that a shorter circuit results in lower latency. A circuit
with all nodes in the same country as the client should lower latency and
removing the middle relay further so. This expected behaviour is consistent
with the results shown in Figure~\ref{fig:p4}.

Figure~\ref{fig:p5} presents a side-by-side comparison between the default
configuration of \ac{dohot} (using torrc) introduced by Alex
Muffet~\cite{DoHoT}, tuned circuits for \ac{dohot} and \ac{dotor}, as well as
\ac{odoh} with a Swedish proxy. Even though the improved query latency of
\ac{dohot} and \ac{dotor} is worse than \ac{odoh}, we managed to cut the
latency in half compared with the default Tor configuration while still
having a stronger threat model than with single collusion-prone oblivious
proxies.
