\section{Background} \label{sec:background}

\subsection{DNS} \label{sec:bg:dns}
The \acf{dns} is a hierarchical and distributed database that maps domain names
to resource records such as IP addresses~\cite{RFC1034,RFC1035}. It enables
human-readable identifiers such as \texttt{example.com} to be translated into
machine-usable network addresses for communication on the Internet.

When a client needs to resolve a domain name, it issues a DNS query, typically
to a \ac{dns} resolver operated by an \ac{isp} or a public \ac{dns} provider.
The resolver acts on behalf of the client to perform a sequence of lookups
through the \ac{dns} hierarchy before responding with the answer back to the
client. Resolvers often cache responses to improve performance and reduce load
on upstream servers. However, because the resolver is the first party to see
each client query, it occupies a privileged and potentially privacy-invasive
position in the \ac{dns} ecosystem.

\subsection{Encrypted DNS Transport} \label{sec:bg:encr}
Several encrypted transport protocols have been standardized to mitigate the
privacy risks associated with plaintext \ac{dns} queries. These protocols aim
to protect the confidentiality and integrity of \ac{dns} traffic between the
client and the resolver.

\acf{dot}~\cite{RFC7858} encapsulates \ac{dns} queries within a \ac{tls}
session, ensuring that on-path observers cannot inspect or tamper with \ac{dns}
traffic. Similarly, \acf{doh}~\cite{RFC8484} transmits \ac{dns} messages over
HTTPS, allowing \ac{dns} queries to blend in with regular encrypted web
traffic. More recently, \acf{doq}~\cite{RFC9250} has been proposed to reduce
latency and connection overhead by using the QUIC transport protocol, which
integrates \ac{tls} over UDP and offers faster handshakes.

While these protocols protect \ac{dns} queries from third-party passive
eavesdroppers on the network path, they do not address a key structural
vulnerability: the resolver remains fully capable of observing and logging all
\ac{dns} queries. This exposes users to potential privacy risks, including
profiling, surveillance, or data misuse by the resolver operator.

\subsection{Tor} \label{sec:bg:tor}
Tor is a low-latency anonymity network that protects users against surveillance
and traffic analysis~\cite{dingledine2004}. It achieves anonymity by routing
traffic through a series of volunteer-operated relays and applying layered
encryption such that no single relay knows the communication's origin and
destination.

A typical Tor circuit consists of three relays: an entry relay, a middle relay,
and an exit relay. The entry relay is the only node that can observe the client's IP
address. The middle relay forwards encrypted traffic between the entry and exit
without access to either endpoint. The exit relay communicates with the final
destination but does not know the client's IP address.

\subsection{Anonymous Queries} \label{sec:bg:anon}
While encrypted \ac{dns} protocols such as \ac{doh} and \ac{dot} protect query
contents from on-path observers, they do not prevent the resolver from linking
\ac{dns} queries to client identities. Several approaches have been developed
to introduce anonymity into \ac{dns} resolution by explicitly separating the
client from the query.

\ac{odns} achieves this by using its own authoritative namespace where the
authoritative name servers act as recursive resolvers~\cite{ODNS}. A client
encrypts the domain name in their query using a key, which itself is encrypted
using the public key of the name server. and appends the special use \ac{tld}
\texttt{odns}. The client sends the query together with the encrypted key to
any other resolver, which will forward the query to the \ac{odns} name server.
The \ac{odns} name server can decrypt the key used to hide the real domain and
perform a recursive lookup against the authoritative name servers before
encrypting the answer and sending it back to the client via the other resolver.
Because the proxy resolver only observes the ciphertext, it cannot determine
the queried domain, and the \ac{odns} name server has no information about the
client issuing the query. There are two main limitations with \ac{odns}. The
\ac{ecs} is part of the \ac{edns} and allows resolvers to include the client's
subnet when querying authoritative name servers. The purpose of this extra
information is to provide the client with a closer instance of the service if
multiple are available. However, this feature may de-anonymize the client in a
system like \ac{odns}. It is in general recommended to turn \ac{ecs} off for
privacy reasons. Another limitation of \ac{odns} is collusion between the
\ac{odns} name server and the resolver, since they have one piece of the puzzle
each.

\ac{odoh} builds on the idea from \ac{odns} but is instead using the \ac{doh}
transport~\cite{RFC9230}. Clients send encrypted queries to a proxy (the
\ac{doh} forwarder), which relays them to a \ac{doh} server (the target
resolver). The proxy can observe the client's IP address but not the query,
while the target resolver can observe the query but not the client's identity.
This design assumes that the proxy and resolver do not collude, an assumption
that is difficult to verify in practice.  The Mutualized Oblivious
DNS~\cite{kurihara2023muodns} ($\mu$ODNS) is an attempt to minimize the
collusion between the relay and the target resolver and corruption of the
client's privacy. For this purpose,  multiple relays are selected randomly to
be used towards the target resolver.  However, to be the best of our knowledge,
there is a significant lack of global $\mu$ODNS adoption.

The effectiveness of \ac{odns} and \ac{odoh} hinges on the availability of
resolvers and proxies that support the protocols. If this number is limited, it
can hinder the scalability and accessibility of the protocols for end-users.
\ac{odns} needs custom authoritative name servers which also act as recursive
resolvers, and \ac{odoh} needs custom proxies between clients and the recursive
\ac{doh} resolvers. The \ac{odoh} has seen deployment from Cloudflare with
several proxy partners~\cite{singanamalla2020,cfblog}. Cloudflare has also
commented on Mozilla's implementation of \ac{doh} where they suggested that
Firefox also could help with discovering \ac{odoh} proxies which is important
for a widespread adoption of the protocol~\cite{cfmozilla}. 

\ac{dohot} is an alternative method for anonymizing queries introduced by Alec
Muffet~\cite{DoHoT}. By routing \ac{doh} traffic through the Tor network, this
approach enables clients to conceal their identity using Tor's existing relay
infrastructure without requiring additional cryptographic protocols or
specialized infrastructure. The \ac{dns} queries remain encrypted end-to-end
via \ac{doh} and Tor ensures anonymity between the client and the resolver.
Muffett's extensive home testing during the pandemic lockdown demonstrated
that, despite increased latency, the system maintained acceptable performance
for everyday Internet activities.
